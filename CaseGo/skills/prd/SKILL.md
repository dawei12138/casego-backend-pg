# 产品需求文档（PRD）

## 概述

设计全面、生产级的产品需求文档（PRD），用于弥合业务愿景与技术执行之间的差距。该技能适用于现代软件系统，确保需求被清晰定义。

## 何时使用

在以下情况下使用此技能：

- 开始新的产品或功能开发周期
- 将模糊的想法转化为具体的技术规格
- 定义 AI 驱动功能的需求
- 利益相关方需要统一的“单一事实来源（source of truth）”来明确项目范围
- 用户要求“编写 PRD”、“记录需求”或“规划功能”

---

## 操作流程（Operational Workflow）

### 阶段 1：需求发现（Discovery / 访谈）

在编写 PRD 的任何内容之前，你**必须**向用户提问以填补信息缺口。不要假设上下文。

**需要询问：**

- **核心问题（Core Problem）**：为什么现在要构建这个？
- **成功指标（Success Metrics）**：我们如何判断它成功了？
- **约束条件（Constraints）**：预算、技术栈或截止时间？

### 阶段 2：分析与范围界定

综合用户输入，识别依赖关系和隐藏复杂性。

- 绘制 **用户流程（User Flow）**
- 定义 **非目标（Non-Goals）** 以保护项目时间线

## PRD 质量标准

### 需求质量

使用具体、可衡量的标准。避免使用“快”、“简单”或“直观”这样的描述。

```diff
# 模糊（BAD）
- 搜索应该很快并返回相关结果。
- UI 必须看起来现代且易于使用。

# 具体（GOOD）
+ 对于 1 万条记录的数据集，搜索必须在 200ms 内返回结果。
+ 搜索算法在基准评估中必须达到 >= 85% 的 Precision@10。
+ UI 必须遵循 'Vercel/Next.js' 设计系统，并达到 100% Lighthouse Accessibility 分数。
